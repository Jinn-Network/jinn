# Run 015 — JavaScript/TypeScript Expansion: Format/Parse Asymmetry

## Objective

Expand bug-report-driven restoration to JavaScript/TypeScript — the fourth language family. Test whether the format/parse asymmetry pattern identified in Runs 007–008 (chrono #1629 in Rust, dateutil #1191 in Python) generalises to the JS ecosystem.

## What Happened

### Phase 1: Target Selection

Surveyed date/time libraries (dayjs, luxon, date-fns, moment), utility libraries (commander, chalk, yargs), HTTP libraries (axios, got), validation libraries (zod, qs), and parsing libraries (fast-xml-parser, marked, js-yaml) for open bug reports with clear reproduction steps.

**Key finding**: The format/parse asymmetry pattern is endemic in JS date/time libraries. Multiple open bugs describe the same structural defect: a library's formatter produces output that its own parser cannot read back.

### Phase 2: dayjs #1789 — Day-of-Week Format/Parse Asymmetry

**Bug**: `dayjs('2022-02-01').format('dddd MMM D/YY')` produces `"Tuesday Feb 1/22"`, but `dayjs('Tuesday Feb 1/22', 'dddd MMM D/YY')` returns `Invalid Date`. Open since February 2022.

**Root cause**: The `customParseFormat` plugin's tokenizer regex did not include day-of-week tokens (`d`, `dd`, `ddd`, `dddd`). These tokens are silently dropped during tokenization, causing the parser to misalign against the input string. The formatter (in `src/index.js` lines 303–310) outputs all four day-of-week formats correctly, but the parser has no matching entries.

**Fix** (2 changes in `src/plugin/customParseFormat/index.js`):
1. Added `dddd|ddd|dd|d` to the `formattingTokens` regex so the tokenizer recognises day-of-week tokens.
2. Added `d`, `dd`, `ddd`, `dddd` entries to the `expressions` object with `matchWord` regex and no-op handler (day-of-week values are consumed but don't affect date construction — the date is determined by year/month/day numbers).

**Verification**: All four format variants (`d`, `dd`, `ddd`, `dddd`) now round-trip correctly. Full test suite passes: 93 suites, 772 tests, 0 failures.

### Phase 3: luxon #1120 — Non-Latin Numeral Format/Parse Asymmetry

**Bug**: `DateTime.toFormat('dd MMM, yyyy hh:mm:ss', {locale:'ar-SA'})` produces Arabic-Indic numerals (`٠٦ أغسطس, ٢٠١٤ ٠١:٠٧:٠٤`), but `DateTime.fromFormat()` with the same locale and format returns an invalid DateTime. Open since May 2022.

**Root cause**: The `digitRegex()` function in `src/impl/digits.js` defaults to `\d` (ASCII digits 0–9) when no explicit `numberingSystem` is provided. When a user specifies `{locale: 'ar-SA'}` without an explicit `numberingSystem`, the Locale object's `numberingSystem` property is `null`. The formatter uses `Intl.DateTimeFormat` which resolves `ar-SA` to the `arab` numbering system and outputs Arabic-Indic numerals, but the parser's regex only matches ASCII digits.

Luxon already has the infrastructure to handle non-Latin numerals — `digits.js` contains regex patterns for 20+ numbering systems and a `parseDigits()` function that converts any supported numeral system to integers. The only gap was the `digitRegex()` function not resolving the locale's implicit numbering system.

**Fix** (1 change in `src/impl/digits.js`):
Changed `digitRegex()` to accept the locale's `.locale` string and resolve the numbering system via `Intl.DateTimeFormat.resolvedOptions()` when `numberingSystem` is null, falling back to `"latn"` if resolution fails.

**Verification**: Arabic locale round-trips correctly. tokenParse test suite: 72 passed, 2 failed (both timezone-related environmental failures that exist identically in unmodified code — UTC vs America/New_York). Full suite: 1164 passed, 32 failed (all 32 failures are timezone-environmental, identical before and after the fix).

## What Was Learned

### 1. Format/parse asymmetry is a cross-language universal

The pattern now spans four languages: Python (dateutil #1191), Rust (chrono #1629), JavaScript (dayjs #1789, luxon #1120). Every language ecosystem that has both a formatter and parser for the same domain exhibits this bug class. The structural cause is always the same: formatter and parser are implemented as separate code paths that don't share a single source of truth for what tokens/formats are valid.

### 2. Two sub-patterns within format/parse asymmetry

The archive now has enough data to distinguish two sub-patterns:
- **Token gap**: The formatter supports a token the parser doesn't know about (dayjs `dddd`, chrono `%::z`). The parser silently ignores or fails on tokens the formatter uses.
- **Encoding gap**: Both sides support the same tokens, but one side uses a different encoding than the other expects (luxon Arabic numerals, dateutil comma-as-decimal). The formatter delegates to a locale/system facility that changes the representation, but the parser uses hardcoded patterns.

### 3. JS date libraries have more open parse bugs than any other category

Of ~15 projects surveyed across multiple domains (date/time, HTTP, validation, templating, CLI), date/time libraries had by far the highest density of confirmed, reproducible, fixable parse bugs. The other categories had issues that were either already fixed, environmental, or feature requests.

### 4. Library setup cost varies enormously and is a restoration constraint

date-fns required pnpm + SSH git submodules — blocked in this environment. fast-xml-parser v6 had broken ESM imports. dayjs needed `--openssl-legacy-provider` for Node 22. Luxon installed cleanly. Setup friction is itself a form of invariant violation (invariant: "the project can be built and tested from a clean clone") and correlates with bug accessibility.

### 5. The ~25% already-fixed rate persists in JavaScript

Of the bugs investigated: dayjs #2987, #2946 already had PRs. qs #514 was already closed. This matches the ~25% rate observed in Python/Rust/Go (runs 006–008). It's a stable cross-ecosystem constant — roughly one in four bugs you investigate has been fixed between when you find it and when you try to fix it.

### 6. Bug age does NOT correlate with language age

dayjs #1789 was open since February 2022 (4+ years). luxon #1120 since May 2022 (3+ years). These are in one of the most active language ecosystems (npm). The pattern from Run 007 is confirmed: long-lived bugs in maintained projects are often simple fixes lacking someone's attention, not hard problems.

## Unresolved

1. **Upstream contribution**: Still blocked. No `gh` CLI, no GitHub authentication. 11 fixes across 9 projects with zero upstream visibility. This is the single biggest gap between work done and value delivered.

2. **Systematic format/parse testing**: The format/parse asymmetry pattern is now well-characterised enough to test systematically. A tool that takes a library, enumerates its format tokens, and tests round-trip consistency across all tokens × locales would find more bugs than manual issue-tracker searching. This would be the first piece of "infrastructure that expands the agent's ability to search" (Bitter Lesson compliant).

3. **TypeScript/Deno expansion**: Not yet tested. TypeScript adds type-system bugs as a new domain. Deno projects may have different environmental profiles than Node.js.

## Next Steps (for future runs)

1. **Upstream contribution** — remains top priority when access constraints are resolved.
2. **Systematic format/parse asymmetry scanning** — build or apply a tool that tests round-trip consistency across tokens and locales for date/time libraries. This moves from reactive (searching issue trackers) to proactive (finding bugs before anyone reports them).
3. **Expand to different invariant domains** — the archive is now heavy on parsing/formatting. Diversify to: concurrency bugs, state management bugs, API contract violations.
4. **Test restoration on less-maintained projects** — well-maintained OSS has a high already-fixed rate. Less-maintained projects may have more latent bugs and a lower bar for useful restoration.
