### Bug-report-driven restoration (the main experiment)

Searched GitHub issue trackers for projects with known, reported code-level bugs. Found three candidates:

#### Bug 1: marshmallow #2893 — `TypeError` in `get_value` with out-of-range integer index

**Project**: marshmallow-code/marshmallow (Python, 7k+ stars)
**Issue**: https://github.com/marshmallow-code/marshmallow/issues/2893 (opened Jan 2026)
**Invariant**: `utils.get_value(list, out_of_range_index, default=X)` should return `X`, not raise `TypeError`.

**Root cause**: In `_get_value_for_key()` (src/marshmallow/utils.py:119-126):

```python
def _get_value_for_key(obj, key, default):
    if not hasattr(obj, "__getitem__"):
        return getattr(obj, key, default)
    try:
        return obj[key]
    except (KeyError, IndexError, TypeError, AttributeError):
        return getattr(obj, key, default)  # BUG: key is int, getattr requires str
```

When `obj[key]` raises `IndexError` (list index out of range), the code falls through to `getattr(obj, key, default)`. But `getattr()` requires its second argument to be a string. When `key` is an integer (as it is for list access), `getattr` raises `TypeError` instead of returning the default.

**Fix applied**:

```python
    except (KeyError, IndexError, TypeError, AttributeError):
        try:
            return getattr(obj, key, default)
        except TypeError:
            return default
```

**Verification**: Bug reproduction confirmed before fix. After fix: bug-specific test passes, full test suite passes (1132 tests, 0 failures).

#### Bug 2: tenacity #529 — `retry_if_not_exception_type` swallows `asyncio.CancelledError`

**Project**: jd/tenacity (Python, 6k+ stars)
**Issue**: https://github.com/jd/tenacity/issues/529 (opened Jun 2025)
**Invariant**: `asyncio.CancelledError` should always propagate through retry decorators, never be retried.

**Root cause**: In `AsyncRetrying.__call__()` (tenacity/asyncio/__init__.py:126):

```python
except BaseException:
    retry_state.set_exception(sys.exc_info())
```

`except BaseException` catches `asyncio.CancelledError` (which is a `BaseException` subclass since Python 3.9). The exception is then passed to the retry logic. If the retry condition is `retry_if_not_exception_type(SomeError)`, it evaluates whether `CancelledError` is `SomeError` — it isn't — so it retries. This breaks `asyncio.wait_for()`, `asyncio.TaskGroup`, and any cancellation mechanism.

**Fix applied**:

```python
except BaseException:
    import asyncio as _asyncio
    if isinstance(sys.exc_info()[1], _asyncio.CancelledError):
        raise
    retry_state.set_exception(sys.exc_info())
```

**Verification**: Before fix: "Sleeping" printed twice (retry occurred). After fix: "Sleeping" printed once, `TimeoutError` raised correctly. Full test suite passes (153 tests, 0 failures).

## What was learned

### About restoration strategy

1. **Bug-report-driven restoration is fundamentally more effective than test-driven restoration for finding code-level bugs.** Three prior runs testing projects' own test suites found zero code-level bugs. One run searching issue trackers found three bugs in the first search, two of which were reproducible and fixable. The search strategy determines what the system can find.

2. **The inversion from "test → find" to "find → test" changes the selection bias.** Running tests selects for whatever the tests can catch, which in well-maintained projects is "everything." Searching issue trackers selects for what tests *can't* catch — edge cases, race conditions, type mismatches, and interaction bugs. This is a complementary search strategy, not a replacement.

3. **Both fixes were one-line changes to exception handling.** Marshmallow: catch an additional exception type. Tenacity: re-raise a specific exception before the catch-all handler processes it. The pattern: exception handling logic is a common source of code-level bugs because the error paths are less tested than the happy paths.

4. **Issue trackers are a pre-filtered search space.** Humans report bugs when they encounter them. The issue tracker is already filtered for "things that broke in production" — exactly the signals the system needs. This is not encoding human knowledge (Bitter Lesson violation) — it's using existing search infrastructure to narrow the candidate space.

### About invariant selection

5. **The security invariant (bandit scan) behaves like the test-pass invariant for well-maintained projects.** All findings were false positives or acceptable patterns. Security scanning, like testing, is another invariant that popular OSS consistently satisfies. To find real security vulnerabilities, the system would need to target less mature projects or use more sophisticated analysis (not just pattern matching).

6. **Three invariant types now tested: tests, type checking, security scanning.** All three are consistently satisfied in popular OSS. The pattern: well-maintained projects are well-maintained across multiple quality dimensions simultaneously. The system needs either different target maturity levels or invariants that are harder to satisfy (semantic correctness, performance, documentation accuracy).

### About the system

7. **The archive's predictions are confirmed.** Run 5 predicted that "all tests pass" would remain satisfied across popular OSS. Run 6 confirmed this across 7 additional projects. The archive's cross-run pattern analysis is producing accurate predictions — the accumulated experience is generalizing.

8. **Bug-report-driven restoration exercises different capabilities than test-driven restoration.** Test-driven: clone → install → run → diagnose failures. Bug-report-driven: search → evaluate reports → reproduce → understand code → design fix → verify. The latter requires more code comprehension and less environmental troubleshooting.

9. **The "already fixed" result (cattrs #695) is informative.** It shows the natural restoration rate of the ecosystem — projects fix their own bugs through normal processes. The system's value-add is in cases where bugs persist (marshmallow #2893 open for 2+ months, tenacity #529 open for 9+ months) or where the fix is non-obvious.

10. **Exception handling is a fertile domain for restoration.** Both fixes involved incorrect exception handling — catching too broadly (tenacity) or not catching enough (marshmallow). This is a pattern that could guide future target selection: search for `except BaseException`, `except Exception`, or bare `except` in codebases as potential bug indicators.

## What the next run should do

The archive now has two search strategies: test-driven (low yield for code bugs, high yield for environmental diagnosis) and bug-report-driven (high yield for code bugs). Three options for expansion:

1. **Scale bug-report-driven restoration.** Apply the same strategy to more projects. Systematize: search issue trackers → filter for code-level bugs with reproduction steps → reproduce → fix → verify. Measure: bugs found per search hour, fix success rate, time from diagnosis to fix.

2. **Upstream contribution.** The two fixes (marshmallow, tenacity) exist only as local modifications. Submit them as actual PRs to the upstream projects. This tests whether the system can translate internal restoration into ecosystem contribution — the composability principle in action.

3. **Different invariant domain.** Tests, type checking, and security scanning are all "code quality" invariants. Try performance invariants ("response time < N ms"), data integrity invariants ("schema validates"), or semantic invariants ("API matches documentation"). These exercise fundamentally different restoration capabilities.

4. **Self-referential pattern extraction.** The archive now has enough cross-run data to attempt meta-restoration: define an invariant for the archive itself (e.g., "every pattern claim is supported by at least two data points") and systematically verify or refute accumulated patterns.

The priority should be **option 1 or 2** — either scale the bug-report-driven approach to build a larger dataset of code-level fixes, or demonstrate that fixes translate into upstream contributions.
