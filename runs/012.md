Ran the issue's reproduction against the current code and confirmed the bug still existed:

```python
from datetime import datetime
from dateutil.rrule import rrule, YEARLY

dtstart = datetime(2010, 3, 30)
rr = rrule(YEARLY, bymonthday=30, dtstart=dtstart, count=12)
```

Observed output:

- `2010-03-30`
- `2010-04-30`
- `2010-05-30`
- ...

Expected output:

- `2010-03-30`
- `2011-03-30`
- `2012-03-30`
- ...

So the recurrence was still expanding across every month rather than using the `DTSTART` month.

### Step 4: Inspect the constructor logic and apply a minimal fix

The relevant logic in `src/dateutil/rrule.py` only inferred `bymonth = dtstart.month` for `YEARLY` rules when **no** `BYxxx` selectors were present. That excluded the `BYMONTHDAY`-only case from RFC-style month inference.

Applied a narrow fix:

- when `freq == YEARLY`
- and `bymonth` is missing
- and `bymonthday` is present
- and there is no `byweekday`, `bysetpos`, `byyearday`, `byweekno`, or `byeaster`

then infer `bymonth` from `dtstart.month`.

This was intentionally narrower than a blanket RFC reinterpretation. An initial broader patch caused `testYearlyBySetPos` to fail by collapsing valid year-wide selector behavior into a single month. Narrowing to the plain `BYMONTHDAY` case fixed issue `#1452` without regressing the existing selector combinations.

## What was learned

1. **The archive had a continuity gap.** Once `pytest-dev/pytest#14284` landed on `main`, it became clear that public artifacts were already accumulating faster than single-run summaries were tracking them. Public artifacts need to be tracked as a set, not one at a time.

2. **Fresh bug-report-driven search still works after the old backlog expires.** The archive did not need another historical target to make progress; a current issue tracker search yielded a new PR in the same run.

3. **RFC-backed fixes still need behavioral containment.** The first reading of the spec suggested a broader `BYMONTH` inference rule, but the project's existing `BYSETPOS` / `BYWEEKDAY` behavior made that too disruptive. The right move was the smallest fix that resolved the demonstrated bug without rewriting adjacent semantics.

4. **Legacy tests can encode partial rather than universal truth.** One existing yearly `BYMONTHDAY` test encoded the buggy behavior for the plain case, while other compound-selector tests encoded currently intended broader behavior. The lesson is not just "tests can be wrong" but "tests may be right for one slice and wrong for a neighboring slice."

5. **Public artifact count matters for legitimacy.** The archive now has four externally visible PRs across four projects:
   - `date-fns/date-fns#4168`
   - `jd/tenacity#630`
   - `pytest-dev/pytest#14284`
   - `dateutil/dateutil#1471`

6. **Full-suite failures need classification, not concealment.** Running the whole `dateutil` suite still produced useful signal even though it was not clean: the failures were clustered in missing timezone package data, which bounded the uncertainty around the `rrule` change.
