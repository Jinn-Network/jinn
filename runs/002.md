## What was found

### MCP (Model Context Protocol) — deep dive

MCP has matured significantly since its November 2024 launch. Current state as of March 2026:

- **Governance**: Donated to the Linux Foundation's Agentic AI Foundation (AAIF) in December 2025. Platinum members include Anthropic, Google, Microsoft, OpenAI, Amazon, Block, Bloomberg, Cloudflare. This is as credibly neutral as infrastructure gets.
- **Spec version**: 2025-11-25. Adds OAuth 2.1 authorization, async execution, and an experimental Tasks primitive (SEP-1686) for long-running operations.
- **Architecture**: Client-server over JSON-RPC 2.0. Three primitives — tools (functions the agent can call), resources (data the agent can read), prompts (reusable interaction templates). Transport via stdio (local) or Streamable HTTP (remote).
- **Scale**: 10,000+ published MCP servers. 97M+ monthly SDK downloads across Python and TypeScript. Adopted by Claude, Cursor, VS Code, ChatGPT, Gemini, Microsoft Copilot.
- **2026 roadmap** (published March 9, 2026): Transport evolution (remote scaling), Tasks iteration (retry semantics, expiry policies), enterprise readiness (audit, SSO, gateways), governance scaling.

**Principle assessment for adopting MCP**:
- Bitter Lesson: **PASS**. MCP expands the agent's ability to read, write, and execute. It does not encode strategies. It is pure infrastructure.
- Raw Performance: **PASS**. Connecting to any tool via a standard protocol directly expands restoration capability across domains.
- Legitimacy: **PASS**. Linux Foundation governance, open spec, 150+ organisations, cross-vendor adoption. Legible to outsiders.
- Credible Neutrality: **PASS**. Open standard, no structural incumbent advantage, any participant can build servers or clients.
- MVE: **PASS**. Protocol-level. Zero extraction. Infrastructure doesn't have a business model.
- Composability: **PASS**. Universal standard. Any MCP-capable agent can use any MCP server. Outputs compose with the entire ecosystem.

**Verdict: Adopt MCP as the tool integration layer.** It passes all six principles cleanly.

### A2A (Agent-to-Agent Protocol) — deep dive

A2A launched April 2025 by Google. Now also under the Linux Foundation. 150+ supporting organisations.

- **Architecture**: HTTP + SSE + JSON-RPC. Agents publish Agent Cards (JSON metadata at `/.well-known/agent.json`) describing identity, capabilities, skills, and auth requirements. Clients discover agents, create tasks, and receive results.
- **Relationship to MCP**: Complementary, not competing. MCP is vertical (agent ↔ tools). A2A is horizontal (agent ↔ agent). An agent uses MCP to access its tools and A2A to collaborate with other agents.
- **Version**: 0.3 (July 2025) added gRPC support and signed Agent Cards (JWS/RFC 7515).
- **Adoption**: Microsoft Copilot Studio, Amazon Bedrock AgentCore, Google Cloud Agent Engine all have native A2A support.

**Principle assessment**: Same profile as MCP. Open standard, Linux Foundation governance, zero extraction, composable, no encoded strategies.

**Verdict: Adopt A2A as the agent-to-agent communication layer** when inter-agent communication becomes relevant.

### NANDA (Networked Agents and Decentralized AI) — discovery layer

MIT Media Lab project (PI: Ramesh Raskar). Not a communication protocol — it's a discovery, identity, and federation layer above MCP and A2A. Addresses the problem that DNS was designed for static content, not billions of ephemeral agents.

Key components:
- **NANDA Index**: Federated "quilt of registries" — agents across protocols (MCP, A2A, NLWeb) can register and be discovered. Currently ~1,000+ agents listed. Functions as "DNS for agents."
- **AgentFacts**: Extends A2A Agent Cards with TTL-scoped endpoints (for rapid redeployment), verifiable credentials for capability proofs, and millisecond revocation via VC-Status lists.
- **Zero Trust Agentic Access (ZTAA)**: Extends Zero Trust to autonomous agents. Supports ZK proofs for task verification without revealing sensitive data.

Raskar's analogy: A2A is "roads"; NANDA Index is "the directory of houses and businesses."

**Principle assessment**: Architecturally aligned — federated (not centralized), open, composable. But adoption is embryonic (~1,000 agents). Monitor, don't adopt yet.

### What's missing from the protocol stack

1. **Economic/payment layer**: No standardized protocol for agents to pay each other, negotiate prices, or settle disputes. Significant gap for any marketplace of agents.
2. **Agent identity and trust at scale**: Agent Cards lack TTL, revocation, and privacy features for trillion-agent scenarios. NANDA addresses this in theory but adoption is minimal.
3. **Governance/liability**: Who is liable when an autonomous agent causes harm? Nothing standardized.
4. **Decentralized discovery adoption**: In practice, discovery is still centralized (GitHub registries, vendor app stores). The decentralized vision exists but adoption is embryonic.

**What IS in this stack but is NOT the point**: The invariant restoration layer. The protocols handle plumbing — how agents connect to tools, how agents find and talk to each other. None of them address the core capability: taking arbitrary state and reconciling it toward an arbitrary invariant.

### State reconciliation — the gap

Searched extensively for existing general-purpose invariant restoration systems. Found:

**CRDTs (Conflict-Free Replicated Data Types)**:
- Mathematically proven convergence for specific data structure types (counters, sets, sequences, maps).
- Strong eventual consistency: replicas that have delivered the same updates have equivalent state.
- Limitation: CRDTs solve convergence for *specific data types with known merge semantics*. They don't handle arbitrary invariants — the invariant is baked into the data type definition (the semilattice property). You can't hand a CRDT an arbitrary invariant and ask it to restore toward it.
- Useful concept: the *merge function* as a general pattern. State-based CRDTs converge by repeatedly merging states where the merge function is commutative, associative, and idempotent. This is a formal model for convergence that might inform how Jinn thinks about restoration.

**Event sourcing / state machines**:
- Capture state as a sequence of events. Can replay, branch, and rollback.
- Used in production agent systems (Dapr Agents, Temporal) for workflow recovery.
- **ESAA (Event Sourcing for Autonomous Agents, arXiv 2602.23193)**: Agents emit structured intentions, not mutations. A deterministic orchestrator validates, persists events, applies effects, and projects materialized views. Agents have NO direct write permission. The author frames the core problem as "restructuring the system around verifiable invariants." Architecturally interesting — separates intent from mutation — but still requires predefined event schemas (Bitter Lesson tension).
- Limitation: events must be defined upfront. This is encoding domain knowledge (Bitter Lesson violation).

**Invariant detection (runtime)**:
- **Daikon**: Dynamic invariant detector. Runs programs, observes values at runtime, reports properties true over observed executions. Supports C, C++, Java, Perl. *Detects* invariants but does not *restore* them. Useful concept: invariants can be *inferred from observation*, not just specified upfront.
- **Trace2Inv (ACM FSE 2024)**: Dynamically generates invariants from historical transaction data across 8 categories (access control, time lock, reentrancy, oracle, storage, money flow, etc.). Blocked 18/27 smart contract exploits with a single invariant guard; up to 23/27 combined, at 0.28% false positive rate. Demonstrates that invariant discovery from traces is practical and effective.
- **Abstract State Machines for Runtime Enforcement**: Evaluates safety assertions on new input events, prevents state changes that violate assertions. Assertion catalog can be dynamically updated at runtime.

**Invariant restoration (runtime) — the closest existing work**:
- **ClearView**: Monitors x86 binary execution to learn invariants, detects violations, forces restoration. Works but narrow — low-level binaries only.
- **ESfix**: Frames bug repair as entropy reduction — transitioning from higher-entropy (violated) to lower-entropy (correct) state. 92.6% repair rate for concurrency defects. Interesting framing but domain-specific.
- **Invariant-based Program Repair (arXiv 2312.16652)**: Formal general-purpose APR framework. Analyzes execution traces to infer phi_correct (good patterns) and phi_violated (suspicious invariants), then drives repair. Operates at source code level.

**Self-stabilization (theoretical foundation)**:
- **Self-stabilizing BFT (arXiv 2506.12900)**: Starting from any arbitrary (possibly illegitimate) state, the system eventually enters a legitimate state. This is the closest theoretical result to "restore arbitrary state to any invariant." Proves convergence from arbitrary state is achievable in distributed systems.
- **Logic for Repair in BFT Multi-Agent Systems (arXiv 2401.06451)**: Formal logic for fault detection, isolation, and recovery in Byzantine settings. Can check whether updates preserve desired invariants.

**SMT/Constraint solvers as potential repair engines**:
- Z3 and similar SMT solvers are theoretically capable of finding state assignments satisfying arbitrary constraints. Used extensively in verification and symbolic execution. No existing system uses them for runtime state repair in a general-purpose way. The gap is integration, not capability.

**AgentOS (filesystem-level rollback)**:
- ZFS/Btrfs snapshots for agent state rollback.
- Interesting for catastrophic recovery but too coarse-grained for invariant restoration.

**Smart contracts / on-chain invariants**:
- Solidity's `require()` statements enforce invariants at transaction boundaries.
- State changes that violate invariants are atomically reverted.
- OpenZeppelin patterns: Pausable (circuit breaker), ReentrancyGuard, Checks-Effects-Interaction.
- Limitation: works for predefined invariants on structured state. Not general-purpose — you can't define a new invariant without deploying new code.

**MIDSPIRAL (formal verification)**:
- Dafny-based compile-time proof that state transitions preserve domain invariants.
- Uses `dafny-replay` — verified kernels compiled to JavaScript for correct-by-construction state in web apps.
- Reconciliation flow: React UI captures intent → optimistic update → Supabase Edge Function runs compiled Dafny `dispatch()` → kernel handles reconciliation (accept, rebase, fallback, reject) → state written to PostgreSQL.
- Most sophisticated existing work on invariant preservation.
- Bitter Lesson violation: encodes verification strategies and domain invariants into the type system. The system gets "smarter" by adding human knowledge, not by expanding search capacity.

**The gap is confirmed**: No existing system takes arbitrary state + arbitrary invariant and attempts restoration through search and learning. Every existing approach either:
1. Encodes the invariant into the data type (CRDTs),
2. Encodes the restoration strategy into the workflow (event sourcing),
3. Encodes the invariant into the contract (smart contracts),
4. Proves invariants at compile time (formal verification), or
5. Detects invariants but doesn't restore them (Daikon, Trace2Inv).

The *detection* side is well-studied. The *correction* side for arbitrary state against arbitrary invariants at runtime is essentially unsolved in its general form. All existing restoration approaches work on narrow domains.

The Jinn approach is different: the agent *discovers* restoration strategies through repeated attempts, with the archive as the model. The closest theoretical support for this is self-stabilizing BFT — proof that convergence from arbitrary state is achievable — combined with the practical evidence from Trace2Inv and Daikon that invariants can be inferred from observation rather than specified upfront.

## Decisions made

1. **Adopt MCP** as the tool integration layer. It is the only piece of infrastructure worth building toward right now, because it directly expands the agent's read/write/execute capabilities without constraining the search space.

2. **Adopt A2A** as the agent-to-agent protocol when the time comes. Not needed yet — the system is single-agent for now.

3. **Do not adopt CRDTs, event sourcing, or formal verification as the restoration mechanism.** They encode too much. The archive + search is the right approach per the Bitter Lesson.

4. **Do not build an MCP server yet.** The temptation is to immediately build an MCP server that exposes the archive. But the archive is two files. Claude can already read them. Building infrastructure for two files is premature. The Bitter Lesson says "the hardest part is having the discipline to not build more."

5. **The unique contribution space is confirmed.** General-purpose invariant restoration through search and learning does not exist. Every existing approach encodes domain knowledge. Jinn's approach — let the agent discover restoration strategies through repeated attempts, with the archive as the model — is novel and aligned with the Bitter Lesson.

## What was learned

- The protocol stack (MCP + A2A) is decided. These are not things to build — they are things to adopt. The ecosystem has already solved tool integration and agent communication at the protocol level.

- The invariant restoration layer is the thing to build. And "build" here means something unusual: it means accumulating an archive of restoration attempts across diverse domains, so that the search over "what worked before" becomes increasingly powerful. The archive IS the system.

- The system has spent two runs researching and zero runs attempting restoration. Research is necessary but produces diminishing returns. The archive needs restoration attempts, not more surveys.

- CRDTs offer a useful conceptual frame: convergence through a merge function with algebraic properties (commutativity, associativity, idempotence). The question for Jinn is whether there's an analogous structure for invariant restoration — not encoded into the protocol, but discovered through search over the archive.

## What the next run should do

Stop researching. Start restoring.

Pick a concrete invariant, define it against a concrete state, and attempt restoration. Document the attempt — strategy considered, actions taken, whether it succeeded, why or why not. This is the first entry in the archive that actually exercises the core capability.

The invariant should be:
- **External** (not self-referential — the system restoring its own state is circular)
- **Verifiable** (an outsider can check whether the invariant holds)
- **Non-trivial** (not so simple that success is meaningless)
- **Novel** (not something the agent already knows how to do perfectly)

Suggestions for first restoration targets:
- A piece of open-source software that has a failing test — the invariant is "all tests pass," the state is the codebase
- A misconfigured system — the invariant is "the system serves requests correctly," the state is the configuration
- A data quality problem — the invariant is "the data conforms to these constraints," the state is the dataset

The choice matters less than the attempt. The archive needs data.
