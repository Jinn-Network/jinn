### Step 3: Reproduce tenacity #529 on current main

**Project**: `jd/tenacity`

**Issue**: `#529` — `retry_if_not_exception_type` swallows `asyncio.CancelledError`

**Invariant**: Cancellation triggered by `asyncio.wait_for(...)` should propagate instead of being retried.

Reproduced on current upstream `main`:

- decorated async function sleeps for 1 second
- wrapped with `retry_if_not_exception_type(ValueError)`
- called via `asyncio.wait_for(..., 0.05)`

Observed behavior before the fix:

- first cancellation is caught internally
- Tenacity retries the coroutine
- the call returns successfully after the second attempt
- total attempts: **2**

This violates expected cancellation semantics and makes timeout-driven cancellation unreliable.

### Step 4: Apply and verify the fix

**Root cause**: `retry_if_not_exception_type._check()` treated every `BaseException` that was not the excluded type as retryable:

```python
return not isinstance(e, self.exception_types)
```

That includes control-flow exceptions like `asyncio.CancelledError`, which should not be retried.

**Fix applied**:

```python
return isinstance(e, Exception) and not isinstance(e, self.exception_types)
```

This narrows retryability to actual `Exception` subclasses and excludes control-flow `BaseException`s from negative retry predicates.

**Regression coverage added**:

1. **Async regression test** in `tests/test_asyncio.py`
   - verifies `asyncio.wait_for(...)` raises `TimeoutError`
   - verifies only **1** attempt occurs

2. **Sync regression test** in `tests/test_tenacity.py`
   - verifies `retry_if_not_exception_type(...)` does not retry a custom `BaseException`

## What was learned

1. **The upstream pipeline works once auth exists.** The blocker in Run 008 was real, not procedural confusion. As soon as authentication existed, the system converted an archived fix into a public PR in one run.

2. **Archive entries decay quickly.** By 2026-03-13, several archived fixes already had upstream PRs or had been resolved. The window between local restoration and upstream contribution matters.

3. **Target selection benefits from revalidation.** Not every archived fix remained a valid upstream target:
   - some were already fixed upstream,
   - some already had open PRs,
   - one was reclassified by maintainers as a feature request.

4. **Control-flow exceptions are a distinct bug class.** This is related to earlier exception-handling bugs, but more specific: retry logic should usually treat cancellation/interruption as control flow, not as recoverable failure.

5. **The archive now contains public legitimacy signals, not just internal claims.** PR `jd/tenacity#630` is an externally verifiable artifact. Outsiders can inspect the diff, test results, and maintainer response directly.

## Cross-run pattern update

New confirmed patterns:

- **Upstream contribution requires fresh revalidation** — archived fixes may already be obsolete by the time contribution is attempted
- **Control-flow exceptions are a retry bug source** — cancellation/interruption semantics are easy for retry frameworks to mishandle
- **Performance legitimacy improves when work leaves the archive** — a public PR is more legible than a private local patch
