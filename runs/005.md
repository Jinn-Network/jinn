### Observation: tool mismatch as a failure class

mathjs was initially run with Jest (which happened to be available) instead of Mocha (the project's actual test runner). This produced 22 "failing" test suites:

- 5 tests failed because `simplifyConstant` typed signature doesn't accept strings — the test calls `math.simplifyConstant(expr)` where `expr` is a string. But running with Mocha (the correct runner), all 6642 tests pass.
- Multiple test suites "failed to run" because they used Mocha's `before`/`after` hooks (not available in Jest, which uses `beforeAll`/`afterAll`).
- 1 test failed because Node.js 22's `assert.throws` handles Error objects differently under Jest's module transform.

All of these "failures" disappeared when using the correct test runner. **The invariant was never actually violated — the measurement tool was wrong.**

This is a new failure class not seen in Runs 3-4: **tool mismatch**. The violation appeared real in the measurement but was an artifact of using the wrong instrument. This is analogous to a doctor using the wrong diagnostic test and concluding the patient is sick.

### What was found

**Initial state**: 18 type errors, 2 warnings across `fastapi/`

#### Error taxonomy

**Category 1: Missing optional dependencies (4 errors — environmental)**

- `fastapi_cli.cli` not installed
- `python_multipart` not installed
- `email_validator` not installed
- `orjson` not installed

These are optional dependencies that FastAPI imports conditionally. Not installed in the test environment. Same class as Runs 3-4: environmental assumptions.

**Category 2: Type system disagreements (9 errors — structural)**

- Private import usage (`GetJsonSchemaHandler` from `pydantic._internal`)
- `str` vs `JsonSchemaMode` type mismatch in schema generation
- Property override incompatibilities (`content_type`, `model`, `scheme`)
- Assignment type mismatch (`EmailStr`)

These represent deliberate design choices that conflict with pyright's strict rules. FastAPI uses mypy, not pyright, and these patterns are accepted under mypy's type system. These are **invariant definition conflicts** — the invariant (zero pyright errors) doesn't match the project's own type checking standard (mypy).

**Category 3: Possibly unbound variables (5 errors — latent code bugs)**

These are the most interesting findings:

**3a. `cm` possibly unbound** (`fastapi/dependencies/utils.py:583`)

```python
async def _solve_generator(*, dependant, stack, sub_values):
    assert dependant.call
    if dependant.is_async_gen_callable:
        cm = asynccontextmanager(dependant.call)(**sub_values)
    elif dependant.is_gen_callable:
        cm = contextmanager_in_threadpool(contextmanager(dependant.call)(**sub_values))
    return await stack.enter_async_context(cm)  # cm unbound if neither condition true
```

If `dependant` is neither async gen nor gen callable, `cm` is never assigned → `UnboundLocalError` at runtime. The function's callers likely guarantee one condition is true, but the function itself doesn't enforce this. A latent bug: currently unexploitable due to calling conventions, but fragile under refactoring.

**3b. `status_code` possibly unbound** (`fastapi/openapi/utils.py:356,374,394,409`)

```python
if route.status_code is not None:
    status_code = str(route.status_code)
else:
    # ... inspection logic ...
    status_code_param = response_signature.parameters.get("status_code")
    if status_code_param is not None:
        if isinstance(status_code_param.default, int):
            status_code = str(status_code_param.default)
# status_code used here — unbound if route.status_code is None AND
# (status_code_param is None OR status_code_param.default is not int)
operation.setdefault("responses", {}).setdefault(status_code, {})[...]
```

The code has a TODO comment acknowledging this is a hack. `status_code` is used in 4 subsequent locations without a fallback. If `route.status_code` is None and the response class's `__init__` doesn't have an int default for `status_code`, this crashes. Another latent bug: unexploitable with current Starlette response classes (which all have int defaults), but fragile.

## What was learned

### About invariant selection

1. **The invariant "all tests pass" is nearly universal in well-maintained OSS.** 12 projects, 12 passing test suites. The only failures were environmental (timezone, proxy, permissions, missing data) or tool-related (wrong test runner). This invariant is too easily satisfied for popular projects — it doesn't exercise the system's restoration capability.

2. **Different invariants reveal different violation classes.** "All tests pass" finds environmental issues. "Zero type errors" finds latent code defects, tool disagreements, and environmental issues. The invariant choice determines what the system can see. A system that only checks one invariant type has a blind spot.

3. **Invariant and tool must agree.** FastAPI passes its own type checking (mypy) but fails pyright. mathjs passes its own test runner (Mocha) but fails Jest. The invariant must be defined in the target's own terms, not imposed externally. Imposing an external invariant measures the gap between two standards, not the health of the code.

4. **Latent bugs exist below the test surface.** The 5 "possibly unbound" variable errors in FastAPI are genuine code defects that tests don't catch because the preconditions in calling code prevent the buggy paths from executing. Type checkers find bugs that tests miss. This suggests that the most valuable restoration targets are not "tests that fail" but "defects that tests can't see."

### About the search process

5. **Target selection difficulty scales inversely with ecosystem maturity.** Well-maintained projects are well-tested. The more popular and maintained a project is, the less likely it has failing tests. Finding code-level bugs requires either: (a) less mature projects, (b) recently introduced regressions, (c) different invariant types, or (d) edge cases that tests don't cover.

6. **The search over candidates is informative even when no target is selected.** Trying 12+ projects and finding zero code-level test failures is itself a strong signal. It tells the system: "all tests pass" is the wrong invariant for exercising restoration against popular OSS. Adjust the strategy.

7. **Language correlates with environmental sensitivity.** Rust projects (fd, ripgrep) and Go projects (gjson, bubbletea) had zero environmental failures. Python projects were vulnerable to proxy contamination, missing timezone data, and install conflicts. JavaScript projects were vulnerable to timezone settings. Compiled languages with static linking are more environmentally resilient.

### About the system

8. **Three restoration attempts now form a pattern.** Run 3 (httpie): 1 fix applied, 3 environmental failures. Run 4 (requests): 0 fixes applied, 30 environmental failures. Run 5: 0 fixes applied across 12 projects, 0 code-level failures found. The trend: as the system searches more broadly, it finds that most violations in well-maintained software are environmental, not logical. This is useful meta-knowledge, but it also means the archive is accumulating data about environmental sensitivity rather than restoration skill.

9. **The archive needs a strategy shift.** Five runs in, the system has demonstrated: (a) environmental failures are systematic and well-understood, (b) code-level bugs are rare in popular OSS, (c) different invariants reveal different classes of violations. To build restoration capacity (the Raw Performance principle), the next runs should either: target less maintained projects where code bugs exist, target non-test invariants (type safety, security, performance), or target the system's own emerging infrastructure.

10. **Tool mismatch is a meta-failure class.** Running the wrong test tool, using the wrong type checker, or applying the wrong invariant standard produces false violations. The system must verify that the measurement tool matches the target's own standards before diagnosing violations. This is a calibration step that future restoration attempts should include.

## What the next run should do

The archive is saturated on "all tests pass" against popular OSS. Three options for diversification:

1. **Different invariant type**: Target security invariants ("no known vulnerabilities"), performance invariants ("response time < N ms"), or data integrity invariants ("schema validates"). These exercise different restoration skills and produce different failure data.

2. **Different target maturity**: Instead of top-1000 GitHub projects, target less mature projects, recent forks, or projects with open "bug" issues. Code-level bugs are more likely where testing discipline is lower.

3. **Self-referential restoration**: Define an invariant for the jinn system itself and attempt to restore it. This is circular (Run 2 warned against it), but now that the archive has five runs of external data, applying the system's own restoration patterns to itself would test whether the accumulated experience generalizes.

4. **Cross-system restoration from Run 4**: Apply the Mechanism 1 fix (proxy variable fixture cleanup) to psf/requests as an actual upstream contribution. This tests whether the system can translate diagnosis into action across project boundaries.

The priority should be **option 1 or 2** — the archive needs data from invariant types other than "all tests pass," or from targets where code-level bugs actually exist.
