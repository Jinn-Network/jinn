## Bugs investigated

### 1. serde_json #1155 — Untagged enum deserialization with Vec<Value>

**Project:** serde-rs/json (Rust, 4.8k stars)
**Bug:** Deserializing into `Vec<serde_json::Value>` variant of untagged enum fails when a `u128` variant is present.
**Outcome:** Not fixable here. Root cause is upstream serde issue #2230 affecting u128 in untagged enums. Known workaround: use u64 instead.
**Classification:** Upstream dependency bug (same class as the charset_normalizer issue in Run 3).

### 2. tokio #6866 — Implicit 1ms delay in sleep_until

**Project:** tokio-rs/tokio (Rust, 28k stars)
**Bug:** Reporter observed 1ms delay when using `sleep_until` with paused time.
**Outcome:** Not a bug. Maintainer (Alice Ryhl) explained: deadlines are rounded to millisecond "ticks" since runtime start, not durations rounded up. Using `start_paused = true` instead of manual `pause()` avoids the issue.
**Classification:** Documentation/understanding issue, not a code defect.

### 3. tera #897 — Escaping applied at unexpected level

**Project:** Keats/tera (Rust, 3.5k stars)
**Bug:** Auto-escaping during string concatenation in comparisons produces wrong results.
**Outcome:** Already fixed in Tera 2. Maintainer confirmed escaping is now done only when writing to the buffer.
**Classification:** Already resolved (~25% already-fixed rate continues).

### 4. chrono #1629 — `%::z` parsing fails for timezone offsets with seconds ✅

**Project:** chronotope/chrono (Rust, 3.4k stars)
**Bug:** The `%::z` format specifier documents support for timezone offsets with seconds (`+09:30:00`), and correctly formats them, but fails to parse them back with `ParseError(TooLong)`.
**Root cause:** The `timezone_offset()` function in `src/format/scan.rs` only parses hours and minutes, never seconds. After parsing `+01:30`, the remaining `:01` is left unconsumed, triggering `TooLong`.
**Fix:** Added `allow_seconds: bool` parameter to `timezone_offset()`. When true and a colon separator followed by exactly two digits is found (without trailing digit ambiguity), seconds are parsed and added to the total offset. `TimezoneOffsetDoubleColon` passes `allow_seconds=true`; all other callers pass `false`.
**Files changed:**
- `src/format/scan.rs` — Added seconds parsing logic to `timezone_offset()`
- `src/format/parse.rs` — Split match arm so `TimezoneOffsetDoubleColon` passes `allow_seconds=true`
- `src/offset/fixed.rs` — Updated call site signature
- `src/datetime/tests.rs` — Updated 3 tests that asserted buggy behavior
**Verification:** All 287 unit tests + 280 doc tests pass. Round-trip (format then parse) preserves value. Negative offsets with seconds work correctly.

### 5. chrono #1625 — DST earliest/latest swapped in Ambiguous result ✅

**Project:** chronotope/chrono (Rust, 3.4k stars)
**Bug:** When converting a `NaiveDateTime` during DST end to local timezone, `MappedLocalTime::Ambiguous(early, late)` returns the values in wrong order: latest first, earliest second.
**Root cause:** In `src/offset/local/tz_info/rule.rs`, the `find_local_time_type_from_local` method constructs `Ambiguous` with `(self.std, self.dst)` for the `Ordering::Less` case (std offset < dst offset). Since higher offset = earlier UTC time, the correct order should be `(self.dst, self.std)`. All four `Ambiguous` construction sites had the same bug.
**Fix:** Swapped the order in all four `Ambiguous(...)` constructions in `rule.rs`:
- Northern hemisphere DST end: `(std, dst)` → `(dst, std)`
- Southern hemisphere DST end: `(std, dst)` → `(dst, std)`
- Southern hemisphere reverse DST start: `(dst, std)` → `(std, dst)`
- Northern hemisphere reverse DST start: `(dst, std)` → `(std, dst)`
**Files changed:**
- `src/offset/local/tz_info/rule.rs` — Fixed ordering in all four Ambiguous construction sites
**Verification:** All 287 unit tests + 280 doc tests pass. Verified with CET/CEST timezone: `Ambiguous(+02:00, +01:00)` now correctly has +02:00 (00:00 UTC) as earliest and +01:00 (01:00 UTC) as latest.

### 6. viper #2095 — `os.Exit(1)` in goroutine during WatchConfig ✅

**Project:** spf13/viper (Go, 27k stars)
**Bug:** `WatchConfig()` calls `os.Exit(1)` inside a goroutine when `fsnotify.NewWatcher()` fails. This causes silent program termination with no way for callers to handle the error, making CI pipelines fail intermittently (~50% failure rate in Docker reported by the issue author).
**Root cause:** Watcher initialization was placed inside the goroutine with `os.Exit(1)` as the error handler, and a `sync.WaitGroup` was used to synchronize. This design means the caller can never recover from watcher creation failure.
**Fix:** Moved all initialization logic (watcher creation, config file resolution, directory watch setup) out of the goroutine and into the main function body. On any initialization error, the function now logs and returns instead of calling `os.Exit(1)`. The goroutine is only launched after all initialization succeeds, containing only the event loop. This eliminated the need for `initWG` (WaitGroup) entirely, also removing the `sync` import.
**Files changed:**
- `viper.go` — Restructured `WatchConfig()` to move initialization before goroutine launch
**Verification:** All tests pass (`go test ./...`).

## What was learned

1. **Bug-report-driven restoration generalizes across languages.** 3 code-level fixes in Rust and Go, matching the Python success rate from Runs 6-7. The strategy is language-agnostic.

2. **Parsing and timezone bugs are universal.** Both chrono fixes involve parsing (timezone offset parsing) and timezone handling (DST ordering). This mirrors the dateutil fixes from Run 7 — date/time libraries across languages share the same vulnerability class.

3. **Error handling bugs persist across ecosystems.** The viper `os.Exit(1)` bug is the same class as the Python exception handling bugs from Runs 6-7 (marshmallow, tenacity, click). Error handling remains the most common bug category: now 5 of 9 total fixes involve error/exception handling.

4. **Formatting/parsing asymmetry is a specific, reproducible bug pattern.** chrono #1629 is exactly this: the formatter correctly outputs `+01:30:01` but the parser rejects it. This is a new pattern worth tracking — any system with separate format and parse paths can have this asymmetry.

5. **Test suites encoding buggy behavior continues.** chrono had 3 test assertions that verified the parsing bug (asserting that valid input should fail). This is the same pattern as click in Run 7. Tests asserting incorrect behavior may actually indicate long-standing bugs.

6. **~25% already-fixed rate holds.** tera #897 was already fixed in Tera 2. Combined across all runs: cattrs #695, rich #3299, and now tera #897 — 3 of ~12 investigated bugs were already resolved.

7. **Upstream contribution remains blocked.** No GitHub authentication means no PR submission. 9 fixes now exist across 7 projects (marshmallow, tenacity, click, aiohttp, dateutil, chrono, viper) with zero upstream visibility. This is the highest-priority unresolved issue.

8. **Two fixes in one project is feasible.** Both chrono bugs were in different subsystems (parsing vs timezone rules) and didn't interact. Multiple bugs in a single project can be found and fixed independently.
