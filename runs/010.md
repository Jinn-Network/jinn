## Bugs investigated

### 1. dayjs #2955 — String timestamp parsing returns wrong value

**Project:** iamkun/dayjs (JavaScript, 47k stars)
**Bug:** `dayjs("1762232987879").valueOf()` returns `-6503268771000` instead of `1762232987879`. Numeric input works correctly; string input of the same timestamp produces a garbage date.
**Root cause:** The `REGEX_PARSE` regex (`/^(\d{4})[-/]?(\d{1,2})?[-/]?(\d{0,2})[Tt\s]*(\d{1,2})?:?(\d{1,2})?:?(\d{1,2})?[.:]?(\d+)?$/`) matches pure-numeric strings like `"1762232987879"` as date components: year=1762, month=23, day=29, hour=87, min=87, sec=9. These nonsensical values are passed directly to the `Date` constructor without validation, producing a garbage date.
**Fix:** Added an early check in `parseDate()`: if the string is all digits and longer than 8 characters (beyond YYYYMMDD format), treat it as a numeric timestamp by converting to number. The threshold of 8 preserves existing parsing of `"20130108"` as January 8, 2013.
**Files changed:**
- `src/index.js` — Added numeric string detection before REGEX_PARSE matching
**Verification:** Bug reproduction confirmed before fix. After fix: `dayjs("1762232987879").valueOf() === dayjs(1762232987879).valueOf()` returns `true`. Date string parsing (`"20130108"`, `"2025"`, `"2025-01-15"`) all preserved. Full test suite passes (772 tests, 0 failures across all timezones).

### 2. dayjs #2819 — `dayjs.utc()` parsing off by one day for non-ISO date strings

**Project:** iamkun/dayjs (JavaScript, 47k stars)
**Bug:** `dayjs.utc("Fri, Jan 28 2025").format("YYYY-MM-DD")` returns `"2025-01-27"` instead of `"2025-01-28"` when the system timezone has a positive UTC offset (e.g., UTC+2). ISO format strings (`"2025-01-28"`) parse correctly.
**Root cause:** In `parseDate()`, non-ISO date strings that don't match `REGEX_PARSE` fall through to `new Date(date)`. JavaScript's `Date` constructor interprets non-ISO strings in local time. When `utc: true`, the UTC plugin's `init()` extracts UTC components via `getUTCDate()` etc., but the underlying Date object was created in local time. In UTC+2: `new Date("Fri, Jan 28 2025")` creates Jan 28 00:00 local = Jan 27 22:00 UTC, so `getUTCDate()` returns 27.
**Fix:** When a string doesn't match `REGEX_PARSE` and `utc` is true, check if the string lacks timezone indicators (GMT, UTC, +/-offset). If so, re-interpret the local-time components as UTC by extracting them from the `Date` object and constructing via `Date.UTC()`. Strings with explicit timezone info are left unchanged.
**Files changed:**
- `src/index.js` — Added UTC re-interpretation for timezone-ambiguous strings in `parseDate()`
**Verification:** Bug reproduction confirmed with `TZ=Europe/Athens`. After fix: `dayjs.utc("Fri, Jan 28 2025")` returns `"2025-01-28"` in all timezones. Timezone-explicit strings (`"Fri, Jan 28 2025 GMT"`) still parse correctly. Full test suite passes (772 tests, 0 failures). Pre-existing timezone-specific failures (3 in Pacific/Auckland, 3 in America/Whitehorse) remain unchanged — no new failures introduced.

### 3. fast-json-stringify #778 — Multiple `if/then` in `allOf` only evaluates first condition

**Project:** fastify/fast-json-stringify (JavaScript, 3.3k stars)
**Bug:** When a JSON schema uses `allOf` with multiple `if/then` blocks for discriminated unions, only the first condition is evaluated. `stringify({ kind: "obj_2", prop_2: "5" })` returns `{"kind":"obj_2"}` — the `prop_2` property is silently dropped.
**Root cause:** `buildAllOf()` merges all `allOf` items into a single schema using `@fastify/merge-json-schemas`. The merge library nests the second `if/then` inside the first's `then` branch, making it unreachable when the first `if` doesn't match. For `kind=obj_2`: the first `if` (kind=obj_1) fails, falling through to the base schema which has no knowledge of the second condition.
**Fix:** Modified `buildAllOf()` to separate `allOf` items into those with `if/then` and those without. Regular items are merged as before. Items with `if/then` are composed into a chained `if/then/else` structure where each condition is evaluated independently: if the first condition matches, use its `then` schema; otherwise, fall through to check the next condition. This preserves the independent evaluation semantics required by JSON Schema's `allOf` with conditional sub-schemas.
**Files changed:**
- `index.js` — Restructured `buildAllOf()` to handle `if/then` items separately from regular allOf items
**Verification:** Bug reproduction confirmed before fix. After fix: both `{ kind: "obj_1", prop_1: "5" }` and `{ kind: "obj_2", prop_2: "5" }` serialize correctly. Full test suite passes (468 tests, 0 failures, including TypeScript type definitions).

## What was learned

1. **JavaScript expansion confirms language-agnostic strategy.** 3 fixes in JavaScript, matching the 3-fix rate from Rust/Go in Run 8 and Python in Runs 6-7. The bug-report-driven strategy now has consistent results across 4 languages.

2. **Date/time libraries remain the highest-yield targets.** Both dayjs fixes are date/time parsing bugs. Combined across all runs: chrono (Rust), dateutil (Python), and dayjs (JavaScript) — 5 of 12 total fixes are in date/time libraries. This is the most over-represented category by project type.

3. **The REGEX_PARSE ambiguity is the same bug pattern as dateutil.** dayjs #2955 (numeric string matched as date) and dateutil #1191 (comma-separated number matched as decimal) share the same structural vulnerability: a greedy parser matches unintended input because the regex doesn't enforce semantic validity. Parsing heuristics without validation produce silent failures.

4. **UTC interpretation inconsistency is a JavaScript-specific pattern.** dayjs #2819 exists because JavaScript's `Date` constructor has implementation-defined behavior for non-ISO strings — it may interpret as local time or UTC depending on the format. This creates a systemic vulnerability in any JS date library that falls back to `new Date(string)`.

5. **Schema merge libraries can violate JSON Schema semantics.** fast-json-stringify #778 is caused by `@fastify/merge-json-schemas` incorrectly nesting independent `if/then` blocks. The merge operation doesn't preserve the independence semantics of `allOf`. This is a new bug category: semantic violations introduced by schema composition tools.

6. **Two bugs in one project continues to be feasible.** Both dayjs bugs were in the same function (`parseDate()`) but involved different code paths (regex matching vs. `new Date()` fallback). This matches the chrono pattern from Run 8 — multiple bugs in different subsystems of the same project can be found and fixed independently.

7. **GitHub API rate limiting forces alternative search methods.** With 0/60 requests remaining, web search was used instead of the API. This was slower but still effective — 3 bugs selected from ~20 evaluated. The hit rate (3 fixable per ~20 evaluated) is slightly lower than API-driven search (which pre-filters by bug label), but still productive.

8. **Format/parse asymmetry pattern partially confirmed.** dayjs #2819 is a variant of format/parse asymmetry: `dayjs.utc().format()` produces correct output, but `dayjs.utc(string)` parses the same type of input incorrectly depending on timezone context. The asymmetry isn't between format and parse functions directly, but between UTC-mode serialization (correct) and UTC-mode parsing (broken for certain string formats).
