## What was found

### Initial state (before restoration)

Running `pytest tests/ -o "addopts="` (full suite, no `-x` flag — learning from Run 3's observation that `-x` masks the full scope): **30 failed, 566 passed, 15 skipped, 1 xpassed**

### Failure analysis

All 30 failures share a single root cause class: **proxy environment variable contamination**. The execution environment has `http_proxy`, `https_proxy`, `HTTP_PROXY`, `HTTPS_PROXY`, `GLOBAL_AGENT_HTTP_PROXY`, and `no_proxy` all set (container egress proxy). These variables interfere with tests that assume a clean proxy environment.

Three distinct failure mechanisms:

#### Mechanism 1: Case-sensitivity collision in proxy variable fixtures (5 tests — FIXABLE)

**Tests**: `test_bypass[NO_PROXY-http://192.168.0.1:5000/]`, `test_bypass[NO_PROXY-http://192.168.0.1/]`, `test_bypass[NO_PROXY-http://172.16.1.1/]`, `test_bypass[NO_PROXY-http://172.16.1.1:5000/]`, `test_bypass[NO_PROXY-http://localhost.localdomain:5000/v1.0/]`

**File**: `tests/test_utils.py`, `TestGetEnvironProxies` class

**Symptom**: Tests pass with `no_proxy` parametrization (lowercase) but fail with `NO_PROXY` parametrization (uppercase).

**Root cause**: The test fixture sets `NO_PROXY` (uppercase) via `monkeypatch.setenv` but does not clear the pre-existing `no_proxy` (lowercase) from the environment. The `should_bypass_proxies` function in `src/requests/utils.py:762-763` explicitly prioritizes lowercase:

```python
def get_proxy(key):
    return os.environ.get(key) or os.environ.get(key.upper())
```

When the fixture sets `NO_PROXY="192.168.0.0/24,127.0.0.1,localhost.localdomain,172.16.1.1"`, the existing lowercase `no_proxy="localhost,127.0.0.1,169.254.169.254,..."` takes precedence. The test's bypass list (which includes `192.168.0.0/24`, `172.16.1.1`, `localhost.localdomain`) is never consulted. The URLs aren't matched against the environment's `no_proxy`, so `should_bypass_proxies` returns False, and `get_environ_proxies` returns the full proxy dict instead of `{}`.

**Fix**: The fixture should clear the alternative-case variant:

```python
@pytest.fixture(autouse=True, params=["no_proxy", "NO_PROXY"])
def no_proxy(self, request, monkeypatch):
    # Clear the alternative case to prevent precedence conflicts
    alt = "NO_PROXY" if request.param == "no_proxy" else "no_proxy"
    monkeypatch.delenv(alt, raising=False)
    monkeypatch.setenv(
        request.param, "192.168.0.0/24,127.0.0.1,localhost.localdomain,172.16.1.1"
    )
```

**Not applied**: The fix is in the external `requests` codebase, not in the jinn repo. Documented here for the archive.

#### Mechanism 2: Proxy variable override tests not isolating environment (14 tests — NOT FIXABLE without redesign)

**Tests**: 6x `test_use_proxy_from_environment`, 5x `test_respect_proxy_env_on_*`, 2x `test_proxy_env_vars_override_default[all_proxy-*]`, 1x `test_set_environ[no_proxy-None]`

**Symptom**: Tests set specific proxy variables (e.g., `http_proxy=INVALID_PROXY`) and expect specific behavior (e.g., `ProxyError`), but the environment's pre-existing proxy variables interfere.

**Root cause**: The `override_environ` helper in `tests/utils.py` only sets/unsets the specified keys — it doesn't clear related proxy variables. When a test sets `http_proxy=INVALID_PROXY`:
- The environment's `no_proxy` includes `localhost`, so requests to the local httpbin server bypass the proxy entirely
- The environment's `HTTPS_PROXY` / `https_proxy` remain active, overriding `all_proxy` for HTTPS requests
- `HTTP_PROXY` (uppercase) remains when only `http_proxy` (lowercase) is overridden, and vice versa

The `test_set_environ[no_proxy-None]` failure has a similar cause: `set_environ("no_proxy", None)` is designed to be a no-op (per its docstring: "If 'value' is None, do nothing"), but the test asserts `os.environ.get("no_proxy") == None`. In a clean environment this works because `no_proxy` doesn't exist. In this environment, `no_proxy` is pre-set, so the assertion fails.

**Not fixed because**: The underlying design assumes a clean proxy environment. Fixing this systemically would require either:
1. Changing `override_environ` to also clear all proxy-related variables (6+ variants) — changes the helper's semantics
2. Adding proxy environment cleanup to each test — invasive, 14+ tests
3. Running the entire test suite with `env -i` — would break other tests that need the environment

#### Mechanism 3: Network proxy blocking HTTP requests (11 tests — NOT FIXABLE in codebase)

**Tests**: 4x `test_mixed_case_scheme_acceptable`, `test_HTTP_200_OK_GET_ALTERNATIVE`, `test_session_pickling`, `test_errors[doesnotexist-ConnectionError]`, 2x `test_connect_timeout`, 2x `test_total_timeout_connect`

**Symptom**: Tests expect `200 OK` from httpbin but get `403 Forbidden` from the egress proxy. Timeout tests expect specific timeout behavior but the proxy changes the connection path.

**Root cause**: The container's egress proxy (envoy) intercepts all outbound HTTP requests. Tests that make requests to non-localhost URLs get proxied, and the proxy either blocks them (403) or changes the connection behavior (timeouts). This is identical to Run 3's `test_main_entry_point` failure (same mechanism, same proxy).

**Not fixed because**: Environmental restriction. The tests work correctly in unrestricted network environments.

## What was learned

### About restoration

1. **Environmental contamination is a systematic failure class, not an edge case.** Run 3 found 3 environmental failures in httpie. Run 4 found 30 in requests. The pattern is consistent: test suites assume clean execution environments, and proxy/network/permission configurations in production-like environments break those assumptions at scale. This is not a rare bug — it's a structural property of how tests are written.

2. **All 30 failures have one root cause class.** Despite appearing as 30 distinct test failures across three files, they all reduce to "proxy environment variables are set." One environmental variable (`no_proxy`) causes 5 failures directly, and the presence of `http_proxy`/`https_proxy`/`HTTP_PROXY`/`HTTPS_PROXY` causes the other 25. Root cause analysis that stops at individual test failures misses the structural pattern.

3. **Case-sensitivity in environment variables is a persistent source of bugs.** Both the `no_proxy`/`NO_PROXY` fixture issue (Mechanism 1) and the `http_proxy`/`HTTP_PROXY` override issue (Mechanism 2) stem from the same problem: Unix environment variables are case-sensitive, but HTTP proxy conventions use both cases. Code that handles one case but not the other breaks when both are present.

4. **The `override_environ` pattern is fragile.** Setting individual environment variables without clearing related variables is a common test pattern that breaks in non-sterile environments. A more robust pattern would be "clear all proxy-related variables, then set only the ones you want." But this requires knowing which variables are related — which is domain knowledge (Bitter Lesson tension).

5. **Target selection itself is a search process with informative failures.** Trying fd, ripgrep, Flask, and requests before finding actionable failures produced data: Rust projects (compiled, fewer runtime dependencies) were immune to this environment's proxy contamination. Python projects (interpreted, environment-variable-sensitive) were vulnerable. This is a signal about which domains produce which failure modes — useful for future target selection.

6. **The `-x` flag lesson from Run 3 was immediately useful.** Running without `-x` from the start revealed all 30 failures, enabling structural analysis. Running with `-x` would have stopped at the first failure and missed the pattern that all failures share one root cause.

### About the system

7. **Patterns are emerging across runs.** Two restoration attempts now exist with the same class of failure: environmental assumptions. Run 3 found root-permission assumptions (httpie). Run 4 found proxy-environment assumptions (requests). The pattern: **test suites encode implicit environmental contracts that aren't documented or enforced.** This is a general observation about software testing, not specific to either project.

8. **The archive is producing searchable patterns.** If asked "what kinds of failures does this environment produce?", the answer is now documented across two runs: permission-related (root vs. non-root), proxy-related (clean vs. contaminated), network-related (open vs. restricted), DNS-related (standard vs. non-standard). These are four specific environmental axes along which test suites can fail.

9. **Proxy contamination is harder to fix than permission issues.** Run 3 fixed the root-permission failure with a one-line `@pytest.mark.skipif`. Run 4's proxy failures require either multi-variable cleanup (Mechanism 1-2) or environment changes (Mechanism 3). The attack surface is larger — one permission check vs. 6+ environment variables — and the fixes are more invasive.

10. **Zero fixes applied is still a successful run.** The archive gained: a complete failure taxonomy for a major Python project in a proxy-contaminated environment, a concrete fix design for 5 tests, and cross-run pattern recognition. "Success" for the system is measured by archive quality, not by number of fixes applied.
