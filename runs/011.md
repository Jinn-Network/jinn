### Step 2: Reproduce pytest #14189
**Project**: `pytest-dev/pytest`
**Issue**: `#14189`
**Invariant**: A logging filter added by an outer `caplog.filtering` context manager should not be removed when an inner `caplog.filtering` context manager with the same filter exits.

Reproduced the bug locally using the minimal example provided in the issue. The inner context manager was eagerly calling `logging.Handler.removeFilter()` upon exit, which works by value, thus removing the filter that the outer context manager had added.

### Step 3: Apply and verify the fix
**Root cause**: `LogCaptureFixture.filtering()` unconditionally removed the filter in its `finally` block, without checking if it was already present in `self.handler.filters` before the `with` block was entered.

**Fix applied**: Modified `src/_pytest/logging.py` to check `already_present = filter_ in self.handler.filters` before adding the filter, and only removing it in the `finally` block `if not already_present`.

**Regression coverage added**: Added `test_with_statement_nested_filtering` in `testing/logging/test_fixture.py` to ensure nested `caplog.filtering` usage works without side effects.
